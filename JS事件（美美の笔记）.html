✨事件及事件对象✨

一、事件

事件就是一件事情或者一个行为（对于元素来说，它的很多事件都是天生自带的）只要我们去操作这个元素，就会触发这些行为
事件就是元素天生自带的行为，我们操作元素就会触发相关的时间行为

二、事件绑定

给元素天生自带的事件行为绑定方法，当事件触发，会把对应的方法执行（触发某某某事件让它做点事）
目的：给当前元素的某个事件绑定方法（不管是基于DOM0还是DOM2），都是为了触发元素的相关行为的时候能够做点事（也就是把绑定的方法执行）“不仅把方法执行了，而且浏览器还给方法传递了一个实参信息值=>这个值就是事件对象”

（1）DMO级👉事件绑定

box.onxxxx.function(){}
  • 每一个元素对象都是对应类的实例，浏览器天生为其设置了很多私有属性和公有属性的属性方法，而onclick就是其中的一个私有属性（事件类私有属性，还有很多其他事件私有属性）这些属性默认为null
①绑定原理

就是给元素的某一个事件私有属性赋值（浏览器会建立监听机制。当我们触发元素的某个行为，浏览器会自己把属性中的赋值去执行） 
只允许给当前元素的mou’ge某个元素绑定一个方法，多次绑定，后面的方法会代替前面的方法，以最后一个为主

（2）DOM2级👉事件绑定 ⭐

【语法】box.addEventListener('xxx',function(){},false) 
removeEventListener移除 
使用的方法都是EventTarget.prototype定义的 
【语法】box.attachEvent('onxxx',function(){})IE6~8 
dettaheEvent移除
①绑定原理
DOM2事件绑定可以给当前元素的某一个事件

②事件池机制
DOM2基于addEventListener完成事件绑定，是基于“事件池机制”完成的

③兼容问题

[谷歌 VS IE高版本] 
在移除事件绑定的时候，如果移除操作发生在正要执行的方法之前（例如：点击的时候，正要执行FN8，但时在执行FN4的时候，我们把FN8从事件池中移除了），谷歌下是立即移除生效，第一次也不再执行FN8，而IE是在当前本次不生效，下一次点击才生效 
[标准 VS IE低版本] 
标准：addEventListener/removeEventListener 
IE低：attachEvent/dettchEvent
1.this问题
标准浏览器中，行为触发方法执行，方法中的this时当前元素本身，IE低版本this指向window
2.重复问题
标准浏览器中的事件池时默认去重，同一个元素的同一个事件行为不能出现相同的绑定方法，但是IE低版本的事件池机制没有那么完善，不能默认去重，也就是可以给同个元素的同个事件绑定相同的方法了
3.顺序问题

标准浏览器时按照向事件池中存放的顺序一次执行的，而IE低版本时乱序执行，没有规律 
IE低版本浏览器出现的所有问题都是由于本身自带的事件池机制不完整导致的

（3）DOM0和DOM2绑定区别

①机制不同

DOM0采用的是给私有属性赋值，所以只能绑定一个方法 
DOM2采用的是事件池机制，所以绑定多个不同的方法
②移除的操作
DOM2在移除的时候，必须清楚要移除哪一个方法，才能在事件池中移除掉，所以基于DOM2做事件绑定，我们要有“瞻前顾后”的思路，也就是绑定的时候考虑一下如何移除（技巧，不要绑定匿名函数，都绑定实名函数）
③DOM2比DOM0增加无法操作事件行为
DOM2事件绑定中增加了一些DOM0无法操作的事件行为，例如：DOMContentLoaded事件（当前页面中的HTML结构加载完成就执行）

三、元素天生自带事件

（1）鼠标事件

down[按下]鼠标左右键都起作用，它是按下即触发，click[点击]是按下抬起才会触发，而且是先把down[按下]和up[抬起]触发，才会触发click[点击]

click：点击 
PC是点击，移动端是单击。移动端会有300MS延迟的问题 
dblclick：双击 
mouseover：鼠标经过（会冒泡和穿透） 
mouseout：鼠标移出（会冒泡和穿透） 
mouseenter：鼠标进入（不会冒泡 / 穿透） 
mouseleave：鼠标离开（不会冒泡 / 穿透） 
mousemove：鼠标移动 
mousedown：鼠标按下 
mouseup：鼠标抬起 
contextmenu：右键菜单
①mouseenter和mouseover的区别
1.mouseover属于经过事件，从父元素进入子元素，属于离开父元素，会触发父元素的mouseout，触发子元素的mouseover
mouseenter属于进入，从父元素进入子元素，并不算离开父元素，不会触发父元素的mouseleave，触发子元素的mouseenter
2.mouseenter和mouseleave阻止了事件的冒泡传播，而mouseover和mouseout还存在冒泡传播
所以对于父元素嵌套子元素这种情况，使用mouseover会发生很多不愿意操作的事情，此时使用mouseenter会更简单，操作方便，真实项目中mouseenter使用比mouseover多

（3）低版本兼容

在IE低版本浏览器中，浏览器执行绑定的方法，并没有把事件对象传递进来，此时ev===undefined,需要基于window.event来获取（由于是全局属性，鼠标每次操作都会把上一次操作的值替换掉 ）
低版本中没有的属性，手动设置以下，按照自己有的先获取到值，再赋值给标准对应的新属性（经过判断处理后，低版本中也有target/pageX/pageY这些属性了）后期在使用的时候，直接按照高版本的使用即可
box.onclick = function (ev) {
if (!ev) {
ev = window.event
// console.log(ev.srcElement)
//获取事件源 低版本情况下无法直接获取（标准中使用的是ev.target）
ev.target = ev.srcElement
// console.log(ev.pageX)
//低版本浏览器额事件对象中不存在pageX/pageY,所以要手动设置
ev.pageX = ev.clientX +(document.documentElement.scrollLeft || document.body.scrollLeft)
ev.pageY = ev.clientY + (document.documentElement.scrollTop || document.body.scrollTop)
ev.which = ev.keyCode;
//preventDafaule&&stopPropagation低版本下也没有
ev.preventDefault = function () {
ev.returnValue = false
//低版本阻止默认行为
}
ev.stopPropagation = function () {
ev.cancelBubble = true
//低版本阻止冒泡传播
}
}
console.log(ev.target, ev.which)
//手动设置后，可以按照高版本规则来使用
ev.preventDefault();
ev.stopPropagation();
}

六、事件传播机制

触发当前元素的某一个事件（点击事件）行为，不仅当前元素事件行为触发，而且其祖先元素的相关事件行为也会依次被触发，这种机制就是事件的冒泡传播机制

（1）捕获

点击inner的时候首先会从最外层开始向内查找（找到操作的事件源）查找的目的是：构建出冒泡传播阶段需要传播的路线（操作就是按照HTML层级结构找的）

（2）目标阶段（事件源行为触发）

把事件源的相关操作行为触发（如果绑定了方法，则把方法执行）

（3）冒泡传播

按照捕获阶段规划的路线，自内而外，把当前时间源的祖先元素的相关事件行为依次触发（如果某一个祖先元素事件绑定了方法，则把方法执行，没有绑定方法，行为触发了，什么都不做，继续向上传播即可）
①冒泡传播阻止
ev = ev || window.event;
ev.stopProragation ? ev.stopPropagation() : ev.cancelBubble = true

（4）DOM0和DOM2事件

xxx.onxxx=function(){}DOM0事件绑定，给元素的事件行为绑定方法，这些方法都是在当前元素事件行为的冒泡阶段（或者目标阶段）执行的 
xxx.addEventListener('xxx',function(){},false)第三个参数false也是控制绑定的方法在事件传播的冒泡阶段（或者目标阶段执行），只有第三个参数为true才代表让当前方法在事件传播的捕获阶段触发执行（这种捕获阶段执行没啥实际意义，项目中不用）

（5）不同浏览器的外层祖先

不同浏览器对于最外层祖先元素的定义是不一样的： 
谷歌：window->document->html->body.. 
IE高：window->html->body... 
IE低：html->body...

window.event下面是path显示的是捕获规划的路线 
关于事件对象的一些理解： 
1.事件对象是用来存储当前本次操作的相关信息，和操作有关，和元素无必然关联 
2.当我们基于鼠标或者键盘等操作的时候，浏览器会把本次操作的信息存储起来（标准浏览器存储到默认的内存中（自己找不到），IE低版本存储到window.event中）存储的值是一个对象（堆内存），操作肯定会触发元素的某个行为，把绑定的方法执行，此时，标准浏览器会把之前存储的对象（准确来说是堆内存的地址）当作实参传递给每一个执行的方法，所以操作一次，即使再多方法中有ev，但是存储的值都是一个（本次操作信息的对象而已）

七、事件委托（事件代理） ⭐

一个容器中，很多后代元素的点击行为都要处理一些事情，之前的思路是把需要操作的元素一一获取，然后一一做事件绑定，在不同的方法中完成不同的需求：现在不用了，基于事件的冒泡传播，我们可以只给容器的click绑定一个方法，这样不管以后点击的是容器的哪个后代元素，都会通过事件的冒泡传播机制把容器的click行为触发，把绑定的方法执行，我们在方法执行的时候，根据事件对象中的事件源（ev.target）来做不同的业务处理即可，这种机制就是事件委托机制

（1）事件委托应用

1.容器中很多后代元素的某个行为要进行操作，委托给容器处理时不错的选择 
2.元素动态绑定 
3.除了xxx，剩下的元素干同样的事情（此时把点击行为的操作委托给body，事件源时XX做什么，不是统一做什么）
